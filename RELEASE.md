# Release Process

This repository uses a dedicated GitHub Actions release workflow instead of commit-message parsing.

## Overview

Releases are created automatically when changes land on `main`:

1. The workflow builds the action and verifies that `dist/` is up to date.
2. It finds the merged pull request for the pushed commit.
3. It chooses the version bump from PR labels.
4. It creates a GitHub release with GitHub-generated release notes.
5. It updates the moving major tag (`v1`, `v2`, and so on).
6. It runs release cleanup.

This keeps the release flow automated, but removes the fragile dependency on conventional commit messages and PR titles.

## Release Labels

Use one of these labels on the pull request before merging:

| Label | Result |
| --- | --- |
| `release:patch` | Patch release |
| `release:minor` | Minor release |
| `release:major` | Major release |

If no release label is present, the workflow defaults to a **patch** release.

If multiple release labels are present, the workflow uses the highest bump (`major` over `minor` over `patch`).

## Manual Releases

You can also run the `release` workflow manually from the Actions tab and choose `patch`, `minor`, or `major`.

This is useful for:

- direct pushes to `main`
- re-running a release for an emergency fix
- publishing a release when there is no merged PR metadata available

## Release Notes

Release notes are generated by GitHub from merged pull requests and commits. There is no custom changelog parsing logic to maintain.

## Tags

Each release creates a semantic version tag such as `v2.0.5`.

The workflow also updates the matching moving major tag so users can pin to `@v2`.

## Troubleshooting

### Release bump is wrong

- Check the labels on the merged pull request.
- Add `release:minor` or `release:major` before merging when a patch is not sufficient.
- If nothing is labeled, the workflow intentionally falls back to `patch`.

### Release did not run

- Confirm the change reached `main`.
- Check the `release` workflow run in Actions.
- For manual runs, confirm the selected release type.

### Release was already published

The workflow is safe to re-run. If the target tag already exists, it exits without creating a duplicate release.
